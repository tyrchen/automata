# Rust 工作流引擎

(1) 深入分析 n8n 及其主要竞品（如 Zapier, Make, Windmill）的核心功能与架构。研究其工作流的构成元素（触发器、节点、连接）、数据在节点间的传递与转换机制，以及它们如何管理凭证和执行状态。
(2) 探索和评估在 Rust 生态中构建此系统所需的技术栈。包括：
(a) 用于实现高性能异步执行引擎的运行时（如 Tokio）。
(b) 用于解析自定义 DSL 的库（如 nom, pest）或技术。
(c) 用于实现 Web API 和 UI 后端的框架（如 Axum, Actix-web）。
(d) 用于安全存储和管理状态的方案（如使用 SQLx 与数据库交互，或使用 Redis 进行缓存）。
(3) 设计一套简洁、可扩展且对用户友好的领域特定语言 (DSL)。
(a) 确定 DSL 的语法（如 YAML, TOML 或自定义格式），使其易于读写。
(b) 定义 DSL 的核心语义，包括如何声明触发器、节点操作、数据映射、条件逻辑（如 if/else, switch）、循环和错误处理策略。
(c) 在 DSL 设计中专门考虑可测试性，研究如何让用户能为工作流或单个节点定义测试用例。
(4) 构思整个工作流系统的宏观架构。
(a) 设计一个 DSL 解析器，能将 DSL 文本转换为内部的、可执行的图结构（如 有向无环图 DAG）。
(b) 设计一个高并发的工作流执行引擎，负责调度和执行图中的节点，并高效地管理数据流和执行状态。
(c) 规划一个模块化的节点库系统，明确新节点（即与外部服务的集成）的开发和注册流程。
(5) 规划从 DSL 定义生成只读工作流 UI 的实现路径。研究如何将解析后的工作流结构转换为前端友好的数据格式（如 JSON），以便渲染出可视化的流程图。
(6) 综合以上所有研究和设计，撰写一份详细的需求文档，其大纲应包括：
(a) 项目背景与目标。
(b) 竞品分析总结。
(c) DSL 详细设计规范（附带示例）。
(d) 系统架构设计（包括各组件的功能职责）。
(e) 核心功能需求列表（如支持的触发器类型、内置节点、安全机制等）。
(f) 非功能性需求（如性能基准、可伸缩性、安全性考量）。
